## Phase 10 — Dataset Switcher (last feature code)

**Goal:** A user opening the app can see that multiple datasets exist,
which one is active, and switch between them.

**Approach (fixed — do not redesign):**
- Build the switcher from the existing registry. `DATASET_REGISTRY` already
  exposes ids and descriptors (label, kind, review status, provenance).
- UI: a small selector in the panel header area. For each dataset show:
  display label + a badge for kind/status
  (`Synthetic fixture` / `Real AIS — unreviewed`).
- The active dataset must be clearly indicated at all times.
- **Switching = update the `?dataset=` query param and reload the page.**
  This matches the existing boot-once architecture. Do NOT implement
  runtime teardown/rebuild of map, layers, or panels.
- Show short provenance text for the active dataset (source, region, year)
  pulled from the descriptor.

**Out of scope (reject even if tempting):**
- Side-by-side or comparison views
- Any new analysis, thresholds, or metrics
- Restyling unrelated UI
- URL state beyond the existing `dataset` param

**Tests:** unit-test the function that produces the switcher's option list
from the registry (ids, labels, badges, active flag). No DOM tests.

**Acceptance criteria:**
- [ ] Opening `/` shows the switcher with both datasets; active one marked
- [ ] Selecting the other dataset reloads with correct `?dataset=` and renders it
- [ ] Badges correctly distinguish synthetic fixture vs real AIS
- [ ] `npm test` fully green; new unit tests added for option-list builder
- [ ] No changes to analysis.js, measurement-review.js, ais.js logic

**Suggested agent prompt:**
> Read CLAUDE.md and PROJECT_PLAN.md. Implement Phase 10 exactly as
> specified. Propose your file-level plan first before writing code.

## Phase 11 — Evaluation Plan (writing, not code)

**Goal:** Add an "Evaluation Plan" section to README.md that shows how the
system's effectiveness *would* be measured. This is the strongest missing
research-readiness signal for admissions reviewers.

**IMPORTANT — division of labor:** The intellectual content of this section
must be authored by the project owner (with discussion support), NOT
generated by a coding agent. The agent's role in this phase is at most
formatting/integration into README. Reason: this section is the direct seed
of the Statement of Purpose and will be discussed in interviews.

**Content the section must cover (owner drafts):**
- Research question: does the visualization help an analyst distinguish the
  true narrative anomaly (6→7) from over-flagged segments (5→6, 7→8)?
- Task design: e.g., segment classification task with/without the
  redundant-visual-channel encodings
- Measures: identification accuracy, time-to-decision, confidence calibration
- Baseline comparison: same data, table-only or single-channel display
- Why it is not run yet: requires participants + IRB-equivalent process —
  positioned as first-year MSc work
- Threats to validity acknowledged (synthetic data, single scenario)

**Acceptance criteria:**
- [ ] README has an Evaluation Plan section in the owner's words
- [ ] It connects the over-flagging finding to a measurable study design
- [ ] No inflated claims (no "user study conducted", no ML language)
